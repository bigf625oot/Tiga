# QuickModel 改造方案与实施计划

## 1. 现状分析 (Current State Analysis)

基于 `quickModel.md` 文档与当前代码库的对比分析，项目在核心功能上已具备高度一致性（约 90% 符合度），但在部分“智能化”组件和用户体验细节上存在差异。

### 1.1 技术栈概览
- **后端**: FastAPI, SQLAlchemy, Redis, SSE-Starlette
- **前端**: Vue 3, Element Plus, Tailwind CSS
- **Agent 框架**: Agno (Phidata), OpenClaw (Gateway), MCP
- **知识库/RAG**: LightRAG, Graphiti, LanceDB, Pathway
- **搜索**: DuckDuckGo, Tavily, Firecrawl

### 1.2 核心功能对齐情况
| 功能模块 | 现状 | 说明 |
| :--- | :--- | :--- |
| **基础对话 (ChatModel)** | ✅ 已就绪 | 基于 OpenAI/Agno 实现，支持多轮对话。 |
| **联网搜索** | ✅ 已就绪 | 集成 DuckDuckGo 等多引擎。 |
| **知识库 (RAG)** | ✅ 已就绪 | LightRAG + Graphiti 实现图/文混合检索。 |
| **记忆系统** | ✅ 已就绪 | 支持 SQLite/PG 切换，Redis 负责热数据。 |
| **流式响应** | ✅ 已就绪 | SSE 全链路打通。 |
| **文件处理** | ✅ 已就绪 | 支持后台索引及长文档内存兜底。 |
| **引用溯源** | ✅ 已就绪 | 通过 Prompt 约束与后处理实现。 |
| **安全隔离** | ✅ 已就绪 | 检索层已实现 User ID 过滤。 |

### 1.3 主要差距 (Gaps)
1.  **上下文压缩 (Context Compression)**:
    *   **需求**: 使用 Agno 的特定压缩算法 (Summarizer/Compressor)。
    *   **现状**: 目前使用基于 Token 计数的滑动窗口截断 (`get_optimized_context`)。
    *   **影响**: 长对话场景下可能丢失早期关键信息，且无法生成高质量摘要。
2.  **意图识别 (Intent Recognition)**:
    *   **需求**: 使用 Agno 的意图识别模型 (Router)，支持参数提取（实体/时间）。
    *   **现状**: 使用自定义的 `IntentClassifier` (基于关键词/启发式规则)，无参数提取能力。
    *   **影响**: 无法精准提取查询条件（如“上周的会议”），导致检索范围过大或不准确。
3.  **模式切换状态共享 (Shared State)**:
    *   **需求**: Solo 模式与 Quick 模式间无缝切换，状态同步至 Redis。
    *   **现状**: 依赖内存中的 `agent_pool`，重启即丢失，且跨模式切换逻辑未闭环。
    *   **影响**: 用户在 Solo 模式下的操作无法在 Quick 模式中延续，体验割裂。
4.  **检索分级与临时索引 (Retrieval Hierarchy)**:
    *   **需求**: Level 1 (In-memory RAG) -> Level 2 (KB) -> Level 3 (Web) 的自动降级策略；附件支持不入库的临时分析。
    *   **现状**: 附件必须持久化入库才能检索；缺乏明确的 Level 1 临时索引机制。
    *   **影响**: 临时文件分析效率低且污染知识库；无法实现真正的“阅后即焚”。
5.  **前端 PDF 交互 (PDF Interaction)**:
    *   **需求**: 点击引用跳转到 PDF 具体页码/高亮区域。
    *   **现状**: 仅支持 Markdown 片段展示，无内置 PDF 查看器。
    *   **影响**: 溯源体验不完整，无法验证长文档中的具体上下文。

## 2. 改造方案 (Transformation Plan)

### 2.1 核心目标
- **智能化**: 引入 Agno Router 和 Compressor，提升意图理解和上下文管理能力。
- **即时性**: 实现 Level 1 临时索引和细粒度过程反馈，打造“快问快答”的流畅体验。
- **闭环体验**: 打通 Solo/Quick 状态共享，完善从引用到原文的溯源链路。

### 2.2 实施阶段 (4周计划)

#### **第一阶段：核心架构升级 (Week 1-2)**
*目标：夯实智能化与状态管理基础。*
1.  **上下文压缩升级**:
    - [ ] 引入 Agno `Summarizer` 或 `Compressor`，在对话轮次超过阈值时自动触发摘要生成。
    - [ ] 替换现有的 `get_optimized_context` 截断逻辑。
2.  **意图识别增强**:
    - [ ] 引入 Agno `Router` 或轻量级分类模型，替代纯关键词匹配。
    - [ ] 实现参数提取功能（Entity Extraction），从 Query 中提取时间、地点、项目名等实体。
3.  **状态共享重构**:
    - [ ] 将 `agent_pool` 从内存迁移至 Redis，定义标准的 `SharedState` 数据结构。
    - [ ] 实现 Solo -> Quick 模式切换的集成测试，确保 Session ID 和 Memory 的连续性。
4.  **临时索引 (Level 1 RAG) 原型**:
    - [ ] 调研并实现基于 FAISS 内存模式或 LightRAG 临时集合的附件索引方案。
    - [ ] 实现“附件上传 -> 临时索引 -> 对话结束清理”的生命周期管理。

#### **第二阶段：体验与交互优化 (Week 3)**
*目标：提升交互的透明度与溯源能力。*
1.  **细粒度过程反馈**:
    - [ ] 利用 Agno Hooks 机制，自动捕获工具调用状态（如“正在搜索 DuckDuckGo...”、“正在读取 PDF...”）。
    - [ ] 改造 SSE 事件流，透传工具执行进度与中间结果。
2.  **前端 PDF 阅读器集成**:
    - [ ] 集成 `pdf.js` 或类似库，实现 PDF 文件的在线预览。
    - [ ] 实现根据 `pageNo` 和 `chunkId` 自动跳转并高亮原文的功能。
3.  **引用展示优化**:
    - [ ] 统一后端返回的 `structured_refs` 格式，确保包含页码、坐标等定位信息。

#### **第三阶段：验收与文档 (Week 4)**
*目标：确保交付质量与文档一致性。*
1.  **全链路回归测试**:
    - [ ] 覆盖 RAG 检索、联网搜索、文件上传、多轮对话等核心路径。
    - [ ] 验证多租户场景下的数据隔离（User ID 过滤）。
2.  **文档同步**:
    - [ ] 更新 `quickModel.md`，将“使用 Agno 算法”等未落地描述更新为实际采用的技术方案（如“语义摘要策略”）。

## 3. 风险评估
- **技术迁移风险**: 替换 NLU 或压缩算法可能导致短期内回复质量波动，需通过 A/B 测试验证。
- **状态同步复杂性**: 跨模式状态共享涉及前端存储与后端 Session 的协同，需重点测试边界情况（如网络中断后的恢复）。

## 4. 结论
当前系统架构成熟度较高，改造重点在于**深化智能化特性**（压缩与意图）和**完善用户体验细节**。建议按照上述 4 周计划推进，优先解决压缩与状态共享问题。
