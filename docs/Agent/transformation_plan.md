# QuickModel 改造方案与实施计划

## 1. 现状分析 (Current State Analysis)

基于 `quickModel.md` 文档与当前代码库的对比分析，项目在核心功能上已具备高度一致性（约 90% 符合度），但在部分“智能化”组件和用户体验细节上存在差异。

### 1.1 技术栈概览
- **后端**: FastAPI, SQLAlchemy, Redis, SSE-Starlette
- **前端**: Vue 3, Element Plus, Tailwind CSS
- **Agent 框架**: Agno (Phidata), OpenClaw (Gateway), MCP
- **知识库/RAG**: LightRAG, Graphiti, LanceDB, Pathway
- **搜索**: DuckDuckGo, Tavily, Firecrawl

### 1.2 核心功能对齐情况
| 功能模块 | 现状 | 说明 |
| :--- | :--- | :--- |
| **基础对话 (ChatModel)** | ✅ 已就绪 | 基于 OpenAI/Agno 实现，支持多轮对话。 |
| **联网搜索** | ✅ 已就绪 | 集成 DuckDuckGo 等多引擎。 |
| **知识库 (RAG)** | ✅ 已就绪 | LightRAG + Graphiti 实现图/文混合检索。 |
| **记忆系统** | ✅ 已就绪 | 支持 SQLite/PG 切换，Redis 负责热数据。 |
| **流式响应** | ✅ 已就绪 | SSE 全链路打通。 |
| **文件处理** | ✅ 已就绪 | 支持后台索引及长文档内存兜底。 |
| **引用溯源** | ✅ 已就绪 | 通过 Prompt 约束与后处理实现。 |
| **安全隔离** | ✅ 已就绪 | 检索层已实现 User ID 过滤。 |

### 1.3 主要差距 (Gaps)
1.  **上下文压缩 (Context Compression)**:
    *   **需求**: 使用 Agno 的特定压缩算法。
    *   **现状**: 目前使用基于 Token 计数的滑动窗口截断 (`get_optimized_context`)。
    *   **影响**: 长对话场景下可能丢失早期关键信息，虽然节省 Token 但不够“智能”。
2.  **意图识别 (Intent Recognition)**:
    *   **需求**: 使用 Agno 的意图识别模型。
    *   **现状**: 使用自定义的 `NluService` (System Prompt)。
    *   **影响**: 自定义实现可能更灵活，但需确认效果是否优于 Agno 内置方案。
3.  **模式切换状态共享 (Shared State)**:
    *   **需求**: Solo 模式与 Quick 模式间无缝切换，状态同步。
    *   **现状**: 存在 `WorkflowSessionState`，但跨模式切换的“全局状态中心”逻辑尚未经严格验证。

## 2. 改造方案 (Transformation Plan)

### 2.1 核心目标
- **标准化**: 决定是否采用 Agno 原生组件（压缩、意图）或标准化现有自定义实现。
- **稳健性**: 强化模式切换时的状态管理。
- **体验升级**: 提供更细粒度的过程反馈。

### 2.2 实施阶段 (4周计划)

#### **第一阶段：标准化与补齐 (Week 1-2)**
*目标：消除核心功能实现的模糊地带。*
1.  **上下文压缩升级**:
    - [ ] 引入语义摘要模块 (Summarizer)，在对话轮次超过阈值（如 10 轮）时自动触发总结，而非简单截断。
    - [ ] (可选) 评估并集成 Agno 的 `Compressor` 组件。
2.  **意图识别基准测试**:
    - [ ] 对比 `NluService` 与 Agno 内置 Intent Model 在典型场景（RAG, SQL, Chat）下的准确率。
    - [ ] 根据测试结果，决定保留优化 `NluService` 或迁移至 Agno。
3.  **状态共享强化**:
    - [ ] 编写 Solo -> Quick 模式切换的集成测试。
    - [ ] 完善 `WorkflowSessionState` 的序列化/反序列化逻辑，确保从 Solo 模式退出时，Quick 模式能无缝读取任务结果。

#### **第二阶段：体验优化 (Week 3)**
*目标：提升交互的透明度和友好度。*
1.  **细粒度过程反馈**:
    - [ ] 改造 SSE 事件流，增加进度百分比字段 (e.g., `{"step": "indexing", "progress": 40}`).
    - [ ] 前端适配展示进度条或详细步骤日志。
2.  **错误处理人性化**:
    - [ ] 梳理 `agent_execute_step.py` 中的错误码。
    - [ ] 为常见错误（如搜索超时、文件解析失败）提供用户友好的建议文案，而非抛出原始异常。

#### **第三阶段：验收与文档 (Week 4)**
*目标：确保交付质量与文档一致性。*
1.  **全链路回归测试**:
    - [ ] 覆盖 RAG 检索、联网搜索、文件上传、多轮对话等核心路径。
    - [ ] 验证多租户场景下的数据隔离（User ID 过滤）。
2.  **文档同步**:
    - [ ] 更新 `quickModel.md`，将“使用 Agno 算法”等未落地描述更新为实际采用的技术方案（如“语义摘要策略”）。

---

## 3. 风险评估
- **技术迁移风险**: 替换 NLU 或压缩算法可能导致短期内回复质量波动，需通过 A/B 测试验证。
- **状态同步复杂性**: 跨模式状态共享涉及前端存储与后端 Session 的协同，需重点测试边界情况（如网络中断后的恢复）。

## 4. 结论
当前系统架构成熟度较高，改造重点在于**深化智能化特性**（压缩与意图）和**完善用户体验细节**。建议按照上述 4 周计划推进，优先解决压缩与状态共享问题。
