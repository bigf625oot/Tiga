## QuickModel的设计概要：



1. 意图识别 (Intent Recognition)：
在 QuickModel 中，意图识别不只是一个“可选”功能，它是降低延迟（Quick）和节省 Token 成本的关键。
路由决策： 并不是所有问题都需要联网或检索知识库。如果用户只是说“你好”，直接回复即可；如果是问“我昨天的会议记录”，则跳过联网，直奔知识库。
防止工具滥用： 避免模型在不需要的时候调用昂贵的 Graph 检索，减少“幻觉”触发。
推荐实现逻辑：
轻量化预处理： 使用一个极小的模型（或 Agno 的 Router 模式）将用户输入分类为：闲谈(Chitchat)、知识库查询(RAG)、实时搜索(Web)、动作执行(Tool)。
参数提取： 识别出用户想要查询的具体实体（例如：时间范围、特定项目名称），直接传给后续的检索器。
1. ChatModel(快问快答模式)：
   - 基于Agno的base Agent，用于生成对话回复。
   - 输入：用户消息、系统指令、历史对话上下文。
   - 输出：模型生成的回复消息。
2. 支持联网查询（用户自定义开启,程序默认集成DuckDuckGo）：
   - 模型可以通过API调用外部服务（如搜索引擎）获取实时信息。
   - 输入：用户查询（如“当前天气”）。
   - 输出：外部服务返回的结果（如“天气信息”）
3. 支持Agno.tools函数调用（用户自定义开启）：
   - 模型可以调用Agno.tools中的函数执行特定任务（如数据查询、计算等）。
   - 输入：函数名称、参数。
   - 输出：函数执行结果。
   - 模型可以在回复中包含函数调用结果。
4. 长短记忆（用户自定义开启，需要程序侧开启PG或Sqlite）：
   - 模型可以记住之前的对话内容，用于生成上下文相关的回复。
   - 输入：历史对话上下文。
   - 输出：模型生成的回复消息。
5. 知识库集成（默认开启，优先级高，要支持lightRAG或者Grapniti的图谱混合检索）：
   - 模型可以访问和利用知识库中的信息。
   - 逻辑: 默认连接到用户的个人以及有权限的全局知识库。
   - 核心体验：体验: 当用户询问关于“我的文档”或“之前保存的内容”时，Agent 自动检索知识库进行回答。
   - 输入：知识库查询（如“公司介绍”）。
   - 输出：知识库返回的信息（如“公司介绍”）。
   - 检索优先度：
   Level 1: 内存 RAG (当前对话上传的文件) —— 响应最快，关联度最高。
   Level 2: 个人/全局知识库 (Graph + Vector) —— 深度事实。
   Level 3: 联网搜索 —— 补充最新时效性信息。
   如果 Level 1 已经解决了问题，可以按策略停止 Level 2/3 的调用，以实现真正的 "Quick"。

6. 支持附件上传（默认开启）：
   - 模型可以接收用户上传的文件（如文档、图片等）。
   - 输入：用户上传的文件。
   - 输出：模型处理后的结果（如文件分析、图片识别等）。
   - 长PDF/Word -> 自动触发实时索引（In-memory RAG），并提示用户“已学习该文档，你可以开始提问”。
7. 对话过程中增加过程反馈（默认开启）：
   - 模型在处理用户请求时，会提供实时的反馈（如“正在分析文件”、“图片识别完成”等）。
   - 输入：用户请求、模型处理状态。
   - 输出：模型处理反馈（如“正在分析文件”、“图片识别完成”等）。
8. 支持多轮对话（默认开启）：
   - 模型可以处理连续的多轮对话，保持上下文连贯性。
   - 输入：用户消息、系统指令、历史对话上下文。
   - 输出：模型生成的回复消息。
9. 与其它模式共享上下文（默认开启）：
   - 模型可以与其他模式（如chatModel）共享对话上下文，实现跨模式的合作。
   - 输入：其他模式的对话上下文。
   - 输出：模型生成的回复消息。
   - 共享 Session ID： 不同的 Agent 实例指向同一个数据库 Session，这样 Memory 记录是通用的。
   - 状态同步： 如果在 Solo 模式下完成了一个任务，切换回 Quick 模式时，Agent 应该知道任务已完成。这需要一个全局状态中心 (Shared State)。
   - Shared State 中心： 使用 Redis 或数据库存储 State 对象。
   - 关键点： 确保 Instructions（系统提示词）在不同模式下保持一致，否则同一个 Session 下，Agent 的语气和行为逻辑会发生突变。
10. 增加引用溯源（默认开启）：

- 需求描述： 当开启联网查询或知识库检索时，模型生成的回答必须包含引用的来源链接或文档片段。
- 在输出结果中标记 \[1], \[2]，并在回复下方列出“参考资料”列表。
- 利用 Agno 的 references 处理逻辑，将检索到的 context 转换为带元数据的引用。
- 利用 Agno 的 references 时，建议不只是给个链接，而是支持点击跳转到 PDF 的具体页码或行数。
1. 上下文压缩与总结（自动开启）
   - 需求描述： 当对话上下文过长时，模型需要自动压缩上下文，保留核心信息。
   - 实现逻辑： 利用 Agno 的上下文压缩算法，在每次对话结束时，将当前上下文压缩为固定长度的向量表示。
   - 效果： 压缩后的上下文可以用于后续对话，保持上下文连贯性。
2. 意图识别预处理（默认开启）

- 需求描述： 模型在处理用户消息之前，需要先识别用户的意图。
- 实现逻辑： 利用 Agno 的意图识别模型，对用户消息进行分类。
- 效果： 可以根据用户意图，选择合适的执行路径（如查询知识库、调用函数等）。

1. 输出流式响应（默认开启）

- 需求描述： 模型在生成回复时，需要支持流式输出，即逐步返回结果，而不是一次性返回完整回复。
- 实现逻辑： 利用 Agno 的流式输出Server-Sent Events (SSE)接口，将模型生成的回复分块返回。
- 效果： 可以实时展示模型处理过程，提升用户体验。

1. 错误回退机制(Fall Back)

- 需求描述： 当模型在处理用户请求时遇到错误（如无法连接外部服务、函数执行失败等），需要有一个回退机制，确保系统能够继续运行。
- 实现逻辑： 当模型遇到错误时，触发回退逻辑，如返回一个默认回复、通知管理员等。
- 效果： 确保系统在遇到错误时能够正常运行，避免因为错误而中断对话。
- 如果联网搜索失败，Agent 应诚实告知用户并尝试基于已有知识回答。
- 设置 Tool 调用超时限制，避免 Quick 模式变成 "Wait 模式"。

1. 安全与隐私隔离（默认开启）

- 需求描述： 模型在处理用户请求时，需要确保用户数据的隐私和安全。
- 规则： 确保检索时带上 user\_id 过滤标签。
- 实现逻辑： 对用户上传的文件、个人信息等进行加密存储，防止泄露。
- 效果： 确保用户数据在模型处理过程中不被泄露。
